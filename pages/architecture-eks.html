🔄 Pull-Based GitOps

Argo CD inside the cluster polls the Git repo. No external system can push directly to the cluster. This means the cluster auto-heals: drift is detected and reverted within minutes.

🔐 Zero Long-Lived Credentials

GitHub Actions uses OIDC to get a short-lived STS token. The ALB Controller uses EKS Pod Identity. Worker nodes use an IAM role. No AWS_ACCESS_KEY_ID is stored anywhere.

📋 Everything as Code

Terraform provisions all AWS resources: VPC, EKS, ECR, IAM, ALB Controller, and Argo CD. Zero ClickOps. Every change is reviewed via Pull Request and tracked in Git history.

🔀 NAT Gateway

A single NAT Gateway in AZ-a allows private nodes to reach the internet (for ECR pulls, AWS API calls) without a public IP. Managed with an Elastic IP for consistent outbound address.

⚖️ Load Balancer Placement

The ALB Controller places the Application Load Balancer in the public subnets (tagged kubernetes.io/role/elb=1), routing traffic to pods in private subnets.

🔑 EKS Endpoint

EKS control plane is configured with endpoint_private_access=true and endpoint_public_access=true. Internal cluster components communicate without traversing the internet.

Section 04

CI/CD Pipeline

The CI pipeline lives in each application repository (not this GitOps repo). A reference workflow template is provided at examples/gitops-app-repo-ecr-bump.yml.

1
Developer pushes code
A git push to an app repository (BuiHongKong/simple-app-01) triggers the GitHub Actions workflow on the main branch.
2
OIDC token exchange (zero secrets)
GitHub Actions requests an OIDC JWT, then calls sts:AssumeRoleWithWebIdentity to get short-lived AWS credentials for the github-actions-role. No stored keys.
# GitHub Actions step — no secrets stored - uses: aws-actions/configure-aws-credentials@v4 with: role-to-assume: ${{ secrets.AWS_ROLE_ARN }} aws-region: ap-southeast-1
3
Docker build & push to ECR
The image is built and pushed with two tags: stag-v.<SHA> for staging promotion. The ECR repo already exists (created by Terraform).
# Tag format: stag-v.<git-sha> IMAGE_TAG=stag-v.$(git rev-parse --short HEAD) docker build -t $ECR_REGISTRY/$APP_SLUG:$IMAGE_TAG . docker push $ECR_REGISTRY/$APP_SLUG:$IMAGE_TAG
4
Bump image tag in GitOps repo
Using a GITOPS_REPO_PAT_TOKEN, the workflow checks out BuiHongKong/EKS and updates values-staging.yaml with the new image tag, then commits.
# apps/simple-app-01/helm/values-staging.yaml image: tag: stag-v.0.3 ← updated by CI
5
Argo CD detects Git change
Argo CD polls the GitOps repo every ~3 minutes. On detecting the new commit, it renders the Helm chart and runs kubectl apply against the cluster.
6
Promotion to Production
To promote, the image is retagged as prod-v.<SHA> and values-prod.yaml is updated. This can be triggered manually via PR merge or a separate workflow dispatch.
# apps/simple-app-01/helm/values-prod.yaml image: tag: prod-v.0.3 ← promoted from staging

// Only GitHub repos in var.app_repos can assume this role { "Effect": "Allow", "Principal": { "Federated": "oidc:github" }, "Action": "sts:AssumeRoleWithWebIdentity", "Condition": { "StringLike": { "sub": "repo:BuiHongKong/simple-app-0*:*" } } }

GitOps Delivery via Argo CD

6 Argo CD Application resources (3 apps × 2 environments) all point to this repository at branch main. Each uses automated sync with self-heal and prune enabled.

Argo CD Application Manifest (staging)
apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: simple-app-01-staging namespace: argocd spec: source: repoURL: git@github.com:BuiHongKong/EKS.git targetRevision: main path: apps/simple-app-01/helm helm: valueFiles: - values.yaml - values-staging.yaml ← env override destination: namespace: staging syncPolicy: automated: prune: true ← removes deleted resources selfHeal: true ← reverts manual changes syncOptions: - CreateNamespace=true
Sync Behavior
🔄
Auto-sync: Argo CD polls Git every ~3 minutes. Any new commit is applied automatically.
🩹
Self-heal: If someone kubectl deletes a pod or edits a ConfigMap manually, Argo CD reverts it.
✂️
Prune: Resources removed from Git are also deleted from the cluster.
📁
CreateNamespace: Argo CD creates the staging/prod namespaces if they don't exist.

Design Rationale

Choice Why
Terraform numbered files Explicit dependency order visible at a glance. Each file = one layer of concern, easy to locate and modify without a module maze.
Private subnets for nodes Nodes have no public IPs and cannot be reached directly from the internet. Only the ALB (in public subnets) can forward traffic in.
GitOps (pull) vs push CD Git repo IS the backup of cluster state. Disaster recovery = kubectl apply -k argocd/. No external system has cluster credentials.
OIDC / Pod Identity (no keys) Short-lived tokens expire in ≤1h. Nothing to rotate, nothing to leak, nothing to accidentally commit.
ALB Controller via Helm Delegates complex L7 routing, SSL termination, and health check management to a proven AWS-maintained component. Developers just write an Ingress object.
for_each on ECR / apps Adding a new app = 1 line in locals.ecr_repos. No copy-paste of resource blocks.
